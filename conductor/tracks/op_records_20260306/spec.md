# 轨道规范 (Track Specification): 部门每日操作记录与报表过滤

## 1. 背景 (Context)
当前的周末加班管理系统按部门隔离管理员工名单和加班状态。所有的部门在生成报表（如统计页）时都会被列出。用户希望引入一种过滤机制：**只有当天确实进行过操作（录入、修改）的部门，才应出现在当天的汇总报表中。**

## 2. 问题陈述 (Problem Statement)
目前系统无法区分“活跃”部门和“静默”部门。无论一个部门是否实际确认了当天的名单，都会被包含在统计中，这导致报表包含过多无意义的空行或未确认的数据。

## 3. 需求描述 (Requirements)
- **数据追踪：** 系统必须能够记录各部门在特定日期的最后操作时间。
- **“操作”定义：**
    - 添加、删除或修改部门内的员工信息。
    - 更改任何员工在周六或周日的加班/出差状态。
- **报表过滤：** 在汇总统计页生成数据时，需关联“操作记录表”，仅保留在查询日期有操作记录的部门。
- **静默处理：** 若某部门在当天没有任何操作，则其所有加班记录在报表中均不予显示，该部门整体留空或隐藏。

## 4. 架构影响 (Architectural Impact)
- **后端：**
    - 新增 `DepartmentOperation` 模型（数据库表：`department_operations`）。
    - 字段：`id`, `department_name`, `date` (YYYY-MM-DD), `last_updated` (DateTime)。
    - 在现有的 `Staff` 和 `Overtime` 修改 API 中加入钩子（Hook），同步更新操作记录。
    - 修改报表查询 API，引入 Join 或 Filter 逻辑。
- **前端：**
    - 无需重大的 UI 变更，但需要确保相关操作（如状态点击）能触发后端的记录更新。
    - 统计页展示逻辑需适配后端返回的过滤后数据。

## 5. 验收标准 (Acceptance Criteria)
- [ ] 后端数据库能正确记录部门操作。
- [ ] 只有当天操作过的部门才会出现在统计汇总页面。
- [ ] 即使某员工已有持久化的加班记录，但所属部门当天未“操作”确认，该员工也不应出现在当天报表中。
- [ ] 系统兼容性：不破坏现有的 SQLite 数据存储和 FastAPI 异步逻辑。
