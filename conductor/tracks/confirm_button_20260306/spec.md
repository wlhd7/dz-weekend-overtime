# 轨道规范 (Track Specification): 首页确认按钮 - 部门加班名单状态确认

## 1. 背景 (Context)
当前的周末加班管理系统按部门隔离管理加班状态。根据之前的需求（`op_records_20260306`），只有当天进行过“操作”的部门才会出现在统计报表中。然而，如果一个部门的加班名单与上周一致且无需修改，部长仍需一种方式来告知系统“名单已确认，请包含在报表中”，而无需进行无谓的修改操作。

## 2. 问题陈述 (Problem Statement)
如果部长的加班名单没有变化，目前系统无法在不进行修改（如增删改员工或状态）的情况下，标记该部门为“活跃”或“已确认”。这导致如果名单保持不变且没有手动触发操作，该部门将无法出现在统计报表中。

## 3. 需求描述 (Requirements)
- **UI 组件：** 在首页“全部出差”按钮右边（同一行）新增一个“确认”按钮。
    - **样式：** 蓝色加粗边框，白色背景。
    - **交互：** 点击后，按钮状态变为“已禁用 (Disabled)”，并显示“已确认”。
- **确认逻辑：**
    - 点击“确认”按钮应向后端发送请求，记录该部门在当天的操作（更新 `department_operations` 表）。
    - **自动确认：** 如果用户进行了任何其他修改（如修改员工状态、添加/删除员工、点击“全部出差”），系统应自动执行确认逻辑，前端对应的“确认”按钮也应同步变为禁用状态。
    - **范围：** 该确认操作针对整个加班周期（包括周六和周日）。
    - **时效：** 确认状态每日有效一次。即每天第一次进入或刷新后，若未确认且无修改，按钮为可用状态；一旦点击或产生修改，当天剩余时间内该按钮保持禁用。
- **报表集成：** 该确认操作应与 `department_operations` 表同步，确保统计报表能正确包含已确认的部门。

## 4. 架构影响 (Architectural Impact)
- **后端：**
    - 在 `DepartmentOperation` 相关服务中新增一个专用的“确认”接口。
    - 确保新接口能正确记录当前部门在 `department_operations` 表中的操作时间。
- **前端：**
    - 在首页组件中添加新按钮。
    - 页面加载时需查询后端，判断当天是否已存在该部门的操作记录，以决定按钮的初始状态。
    - 任何修改操作完成后，前端应同步更新该按钮为禁用状态。

## 5. 验收标准 (Acceptance Criteria)
- [ ] 首页显示“确认”按钮，位置在“全部出差”右侧，样式符合要求（蓝色加粗边框，白色背景）。
- [ ] 点击“确认”按钮后，按钮立即禁用并显示“已确认”，且后端数据库记录更新。
- [ ] 在页面进行任何其他修改操作后，“确认”按钮自动变为禁用状态。
- [ ] 刷新页面后，若当天已确认，按钮应保持禁用状态。
- [ ] 已确认的部门在统计汇总页面中能正常显示。
